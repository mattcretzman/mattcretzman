## Why Architecture Matters More Than Your LLM

I see teams obsess over which LLM to use—GPT-4 vs Claude vs Llama—while completely ignoring system architecture. This is backwards.

A well-architected multi-agent system with GPT-3.5 will outperform a poorly architected system with GPT-4 every time. Here's why:

**Context window limitations:** Even the best models have limited context. A bad architecture dumps everything into one agent's context, hitting limits and degrading performance. Good architecture distributes context appropriately.

**Latency compounds:** If your agents call each other sequentially, latency stacks multiplicatively. 3 agents × 2 seconds each = 6 seconds minimum. With parallelization, it's still 2 seconds.

**Failure isolation:** When one agent fails in a monolithic design, everything fails. Proper architecture isolates failures and enables graceful degradation.

**Cost scales with architecture:** Sequential workflows make redundant LLM calls. Smart routing and caching can cut costs by 70%.

The LLM is an implementation detail. Architecture determines whether your system works.

## The Four Fundamental Patterns

After building 50+ agents, I've found every multi-agent system falls into one of four patterns—or a hybrid of them. Understanding these patterns prevents you from reinventing wheels that already have known failure modes.

| Pattern | Best For | Fails When | Complexity |
|---------|----------|------------|------------|
| Manager-Worker | Clear task decomposition | Too many workers or ambiguous tasks | Medium |
| Blackboard | Creative/open-ended problems | Tight coupling or time constraints | High |
| Pipeline | Sequential processing | Need for backtracking or branching | Low |
| Peer-to-Peer | Decentralized coordination | Complex consensus requirements | Very High |

Let's examine each in detail.

## Pattern 1: Manager-Worker (And Its Limitations)

The manager-worker pattern is what everyone starts with. One agent (manager) receives the request, breaks it into subtasks, and delegates to specialized worker agents. Workers report back; manager synthesizes the final output.

### When It Works

This pattern excels when:
- Tasks decompose cleanly into independent subtasks
- Workers don't need to communicate with each other
- The manager can verify worker outputs programmatically
- Workload is predictable (not spikey)

### Example: LeadStorm's Content System

Our content generation system uses this pattern:

**Manager Agent:** Receives topic, decides what content types to create (blog post, social threads, email sequence)

**Worker Agents:**
- Research Agent: Gathers data, statistics, quotes
- Writer Agent: Produces draft content
- Editor Agent: Reviews for tone, accuracy, brand voice
- SEO Agent: Optimizes keywords, meta descriptions
- Formatter Agent: Converts to final formats (MDX, HTML, etc.)

The manager coordinates the sequence: research → write → edit → SEO → format. Each worker has one job and reports back when done.

### When It Breaks Down

The manager-worker pattern fails catastrophically in three scenarios:

**1. Too many workers:** With 10+ workers, the manager's context window overflows tracking status. Response quality degrades. I've seen managers start hallucinating worker responses.

**2. Interdependent workers:** When Worker B needs information from Worker C, but Worker C hasn't started yet, you get deadlocks or inefficient polling.

**3. Dynamic decomposition:** If the manager can't predict all subtasks upfront (common in research or creative tasks), it either over-decomposes (wasted work) or under-decomposes (missed requirements).

### Scaling Beyond ~5 Workers

If you need more than 5 workers, add a hierarchical layer. Create sub-managers responsible for clusters of workers. The top manager coordinates sub-managers, not individual workers.

```
Top Manager
├── Content Sub-Manager (Research + Writer + Editor)
├── Distribution Sub-Manager (SEO + Formatter + Scheduler)
└── Analytics Sub-Manager (Tracking + Reporting + Optimization)
```

This two-tier architecture scales to 15-20 workers before hitting new limits.

## Pattern 2: Blackboard Architecture

The blackboard pattern comes from AI research in the 1980s. Multiple agents contribute to a shared workspace (the "blackboard"). No single agent is in charge. Instead, agents opportunistically add, modify, or critique contributions based on their expertise.

### When It Works

This pattern excels for:
- Creative tasks with no single correct answer
- Problems where the solution emerges from iteration
- Scenarios requiring diverse perspectives
- Open-ended research or brainstorming

### Example: OpenClaw's System Design Agent

When OpenClaw designs multi-agent systems for clients, we use a blackboard approach:

**The Blackboard:** A shared document describing the proposed architecture—components, data flows, failure modes, scaling considerations.

**Specialist Agents:**
- **API Designer:** Proposes interface contracts
- **Scalability Engineer:** Identifies bottlenecks and suggests async patterns
- **Security Reviewer:** Flags authentication and authorization gaps
- **Cost Optimizer:** Suggests caching and batching strategies
- **DevOps Planner:** Proposes deployment and monitoring setup

Each agent reads the blackboard, contributes their expertise, and passes control to the next. Multiple rounds iterate toward a robust design.

### Critical Implementation Details

**Contribution rules:** Agents need clear rules about when they can contribute. Without constraints, you get chaos or endless loops.

Our rules:
1. Each agent must make at least one concrete contribution or explicitly pass
2. No agent can contribute twice consecutively
3. If no changes after a full round, terminate
4. Any agent can flag "blocker" issues that require immediate resolution

**State management:** The blackboard grows. Truncate or summarize older iterations to prevent context overflow.

**Termination detection:** Someone needs to decide when the solution is "good enough." We use a Reviewer Agent that evaluates completeness and calls stop.

### When It Breaks Down

Blackboard patterns struggle with:

**Time constraints:** Iterative convergence takes time. Not suitable for real-time applications.

**Conflicting expertise:** When agents disagree fundamentally (e.g., security vs. performance tradeoffs), the blackboard can oscillate without resolution.

**Tight coupling:** If agents depend heavily on each other's outputs, the emergent behavior becomes unpredictable and hard to debug.

## Pattern 3: Pipeline/Fan-Out

The pipeline pattern processes data through sequential stages. Each agent transforms the output and passes it downstream. Fan-out variants split data for parallel processing, then fan-in to combine results.

### When It Works

Perfect for:
- ETL workflows (Extract, Transform, Load)
- Content processing pipelines
- Multi-stage approval workflows
- Data enrichment tasks

### Example: TextEvidence.ai Document Processing

TextEvidence processes legal documents through a pipeline:

```
Raw Upload → Sanitizer → OCR (if needed) → Parser → Analyzer → Formatter → Storage
     ↓
  [Parallel Fan-Out]
     ↓
Entity Extractor → Relationship Mapper → Timeline Builder → Report Generator
```

Each stage is deterministic and stateless. Failed documents are routed to a quarantine queue for manual review.

### Pipeline Optimization Techniques

**Batching:** Process documents in batches of 10-50 to amortize setup costs.

**Parallel stages:** When stages are independent, run them concurrently. Our entity extraction and relationship mapping happen simultaneously.

**Circuit breakers:** If a stage fails repeatedly, skip it and flag for review rather than stalling the entire pipeline.

**Backpressure handling:** When downstream stages are slower than upstream, implement buffering or shed load.

### When It Breaks Down

Pipelines fail when:

**Branching logic is complex:** If Stage A needs to route to Stage B or C based on content, you need a router agent. Too many routers make the pipeline a graph, defeating the simplicity benefit.

**Backtracking required:** When Stage D discovers an issue that requires re-processing from Stage B, you need complex state management.

**Stateful processing:** Agents that need to remember previous documents require external state stores, adding complexity.

## Pattern 4: Peer-to-Peer Mesh

In a mesh architecture, agents communicate directly with each other without central coordination. Each agent decides who to talk to based on the task and current state.

### When It Works

Appropriate for:
- Decentralized systems with no natural coordinator
- Scenarios where agents have equal authority
- Systems requiring high availability (no single point of failure)
- Complex coordination problems like resource allocation

### Example: OpenClaw's Agent Marketplace

We're experimenting with a marketplace where specialized agents advertise capabilities and bid on tasks:

- **Task Agents:** Post jobs with requirements and constraints
- **Worker Agents:** Advertise capabilities and bid on compatible tasks
- **Arbiter Agents:** Resolve conflicts and verify completion

No central manager. Agents negotiate directly using a shared protocol.

### The Consensus Problem

Mesh architectures introduce hard distributed systems problems:

**Consensus:** How do agents agree on shared state? We use a lightweight consensus protocol—agents propose state changes; majority acceptance commits.

**Discovery:** How do agents find each other? We maintain a capability registry. Agents register their skills; others query when needed.

**Byzantine failures:** What if an agent lies or fails silently? Redundancy—critical tasks require multiple agent confirmation.

### When to Avoid

Unless you have a specific reason for decentralization, avoid mesh patterns. They're complex to debug, hard to reason about, and overkill for most applications.

## Communication Protocols That Don't Fall Over

How agents communicate matters as much as the architecture pattern. I've learned this through painful production failures.

### Synchronous vs. Asynchronous

**Synchronous (blocking):** Agent A calls Agent B and waits. Simple to implement. Dangerous at scale.

Problems:
- Cascading failures: If B is slow, A is slow. If B fails, A fails.
- Resource exhaustion: Threads/connections pile up waiting.
- Tight coupling: A needs to know B's location and health.

**Asynchronous (message-based):** Agent A publishes a message; Agent B consumes when ready. Harder to implement. Scales better.

Benefits:
- Decoupling: A doesn't know or care about B's existence
- Resilience: Messages queue if B is down; processing resumes when B recovers
- Load balancing: Multiple B instances share the queue

We use async by default. Synchronous only for latency-critical paths with circuit breakers.

### Message Formats

Standardize your message format. Ours:

```json
{
  "messageId": "uuid",
  "correlationId": "uuid",  // For tracing request chains
  "timestamp": "ISO-8601",
  "sender": "agent-name",
  "recipient": "agent-name | broadcast",
  "messageType": "command | query | event | response",
  "payload": { /* task-specific data */ },
  "context": {
    "conversationId": "uuid",
    "turn": 3,
    "previousMessages": [/* truncated history */]
  },
  "ttl": 300  // Seconds before message expires
}
```

This structure enables:
- **Tracing:** Follow a request through 10 agent hops
- **Debugging:** Reconstruct conversation state
- **Timeouts:** Automatic cleanup of stale messages
- **Retries:** Idempotent processing with messageId deduplication

### Delivery Guarantees

Choose your guarantees based on requirements:

**At-most-once:** Fire and forget. Fastest. May lose messages.

**At-least-once:** Retry until acknowledged. Messages may duplicate. Receivers must be idempotent.

**Exactly-once:** Hardest to implement. Requires deduplication and transaction coordination. Usually not worth the complexity—we use at-least-once with idempotency keys.

## State Management at Scale

Multi-agent systems generate enormous state. Every conversation, every intermediate result, every decision point consumes memory. Managing this state determines whether your system scales or collapses.

### State Categories

**Ephemeral State:** Current conversation context. Discard after task completion. Store in memory or Redis with TTL.

**Working State:** Intermediate results needed across agent boundaries. Store in a shared cache with 1-24 hour TTL depending on workflow length.

**Persistent State:** Final outputs, audit logs, user preferences. Store in your primary database.

### Our State Architecture

```
Agent Memory (Context Window)
    ↓ (summarized)
Conversation Cache (Redis, 1-hour TTL)
    ↓ (key milestones)
Working Store (PostgreSQL, 24-hour TTL)
    ↓ (final results)
Persistent Store (PostgreSQL, permanent)
```

Agents start with their own context. As conversations grow, older context summarizes and moves to cache. Critical milestones checkpoint to working store. Final results commit to persistent store.

### The Context Window Problem

Even with state management, individual agents hit context limits. Solutions:

**Sliding window:** Keep most recent N messages, summarize older ones.

**Hierarchical summarization:** Compress conversation into key points, discard details.

**Selective retrieval:** RAG-style—store conversation in vector DB; retrieve relevant chunks when needed.

We've found sliding window + hierarchical summarization handles 95% of cases. RAG for the remaining 5% with very long conversations.

## Failure Modes and Recovery

Agents fail. Networks partition. LLMs hallucinate. Your architecture must handle this gracefully.

### Common Failure Modes

**Agent crashes:** Process dies, memory lost. Mitigation: Stateless agents with external state. Restart and resume from checkpoint.

**LLM hallucination:** Agent generates invalid output. Mitigation: Validation layers. Schema enforcement. Human escalation for edge cases.

**Infinite loops:** Agents cycle endlessly. Mitigation: Max turn limits. Progress detection ("no meaningful change in 3 rounds").

**Cascading timeouts:** Slow agent backs up the queue. Mitigation: Circuit breakers, bulkheads (isolate slow agents), automatic scaling.

**State inconsistency:** Agents disagree on shared state. Mitigation: Event sourcing (rebuild state from event log), consensus for critical decisions.

### Our Failure Recovery Stack

1. **Retry with backoff:** 3 attempts, exponential backoff (1s, 2s, 4s)
2. **Circuit breaker:** After 5 failures, mark agent unhealthy for 30s
3. **Fallback:** Route to alternative agent or human escalation
4. **Checkpoint recovery:** Resume from last known good state
5. **Dead letter queue:** Failed messages for manual inspection

We measure recovery time from failure detection to full resumption. Target: <5 seconds for automated recovery, <30 seconds for human escalation.

## Choosing the Right Pattern for Your Problem

Here's my decision framework:

### Start Simple

Begin with a single agent. Most "multi-agent" problems are actually single-agent problems with better prompting.

### Add Agents When...

- **Different expertise domains:** One agent can't be expert at everything
- **Scale requirements:** Need parallel processing
- **Failure isolation:** One component failing shouldn't kill everything
- **Team boundaries:** Different teams own different components

### Pattern Selection Matrix

| If your problem is... | Use pattern |
|----------------------|-------------|
| Task decomposition with clear subtasks | Manager-Worker |
| Creative/emergent solution | Blackboard |
| Sequential data processing | Pipeline |
| Decentralized coordination | Peer-to-Peer |
| Mixed requirements | Hybrid (most real systems) |

### Hybrid Example: OpenClaw Platform

OpenClaw itself uses a hybrid architecture:

- **Top level:** Manager-Worker (Orchestrator manages specialized subsystems)
- **Within subsystems:** Blackboard (Design system where specialists collaborate)
- **Data ingestion:** Pipeline (Document processing)
- **Agent marketplace:** Peer-to-Peer (Experimental)

Each subsystem uses the pattern appropriate for its problem domain.

## Key Takeaways

- **Architecture beats LLM:** A well-designed GPT-3.5 system outperforms poorly architected GPT-4
- **Four patterns cover 90% of cases:** Manager-Worker, Blackboard, Pipeline, Peer-to-Peer
- **Async communication scales:** Synchronous calls create cascading failures
- **State management is critical:** Plan for context limits and state persistence upfront
- **Expect failure:** Design for recovery, not perfection
- **Start simple:** Most problems need fewer agents than you think

## What I'd Do Differently

My biggest architectural mistake was over-engineering early. I built a complex peer-to-peer mesh for a problem that needed a simple pipeline. It took weeks to debug and never worked reliably.

Now I follow the "rule of three:" Don't introduce multi-agent complexity until a single-agent solution has failed three times for different reasons. Most problems that seem to need multiple agents actually need:

1. Better prompting
2. Better tools
3. Better state management

Only then consider adding agents.

The second mistake was ignoring operational concerns. Beautiful architectures that are impossible to debug or monitor fail in production. Every design decision gets evaluated on: "Can I understand what happened when this breaks at 3 AM?"
